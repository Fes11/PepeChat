# Архитектура PepeChat

## Структура приложения

### 1. Frontend (Tauri + React)

```
src/
├── api/                    # Сетевой слой
├── components/             # UI компоненты
├── hooks/                  # Бизнес-логика и кастомные хуки
├── store/                  # Глобальное состояние (MobX)
├── services/               # Сервисы для работы с API
└── style/                  # Глобальные стили
```

### 2. Сетевой слой (`api/`)

#### `api/index.jsx` — Axios конфигурация

- Базовый URL: `http://localhost:8000`
- Автоматическая подстановка JWT токена из localStorage
- **Интерцептор запросов**: добавляет `Authorization: Bearer <token>` заголовок
- **Интерцептор ответов**: обрабатывает 401 ошибку, автоматически обновляя токен через refresh token
- Очередь запросов во время обновления токена

#### `api/voiceRoomSocket.js` — WebSocket для голосовых комнат

- Класс `VoiceRoomSocket` для подключения к голосовой комнате
- URL: `ws://localhost:8000/ws/room/{chatId}/`
- Методы: `connect()`, `send(data)`, `disconnect()`
- События: `onOpen`, `onClose`, `onError`, `onMessage`
- Автоматическая сериализация/десериализация JSON

### 3. Хуки (`hooks/`)

#### `hooks/useWebRTC.js` — WebRTC логика

Отвечает за установку P2P соединений между участниками голосовой комнаты.

**Основные функции:**

- `startLocalStream()` — запрашивает доступ к микрофону и создает локальный аудиопоток
- `createPeerConnection(userId, sendSignal)` — создает RTCPeerConnection для пользователя
  - Добавляет локальный аудиопоток
  - Настраивает обработчики `onicecandidate` и `ontrack`
  - Возвращает peer connection
- `handleOffer(data, sendSignal)` — обработка входящего offer'а
  - Создает peer connection
  - Устанавливает remote description
  - Создает и отправляет answer
- `handleAnswer(data)` — обработка входящего answer'а
  - Проверяет signaling state
  - Устанавливает remote description
- `handleIceCandidate(data)` — обработка ICE кандидата
  - Добавляет кандидата в peer connection

**STUN сервер:** `stun:stun.l.google.com:19302`

**Хранение состояния:**

- `peerConnections` (ref) — объект с peer connections по userId
- `localMediaStream` (ref) — локальный медиапоток
- `clients` (state) — массив удаленных участников с их потоками

#### `hooks/useVoiceRoom.js` — Управление голосовой комнатой

Координирует работу WebSocket соединения и WebRTC.

**Основные функции:**

- `connect()` — подключается к голосовой комнате
  - Создает VoiceRoomSocket
  - Настраивает обработчики событий
  - При получении `user_joined` создает peer connection и отправляет offer
- `disconnect()` — отключается от комнаты
  - Останавливает heartbeat
  - Отменяет переподключение
  - Закрывает WebSocket

**Обработчики событий WebSocket:**

- `room_state` — установка начального состояния комнаты (список участников)
- `user_joined` — новый участник
  - Добавляет в список участников
  - Создает peer connection
  - Отправляет WebRTC offer
- `user_left` — участник вышел
  - Удаляет из списка участников
- `webrtc_offer` — входящий offer
  - Вызывает `handleOffer` из useWebRTC
- `webrtc_answer` — входящий answer
  - Вызывает `handleAnswer` из useWebRTC
- `ice_candidate` — ICE кандидат
  - Вызывает `handleIceCandidate` из useWebRTC
- `pong` — ответ на heartbeat

**Heartbeat механизм:**

- Интервал: 10 секунд
- Отправляет `{ type: "ping" }` на сервер
- Автоматическое переподключение при разрыве (задержка 3 секунды)

**Хранение состояния:**

- `participants` (state) — массив участников комнаты
- `socketRef` (ref) — текущий WebSocket
- `reconnectTimeoutRef` (ref) — таймер переподключения
- `heartbeatRef` (ref) — интервал heartbeat
- `manuallyClosedRef` (ref) — флаг ручного отключения

#### `hooks/useMedia.jsx` — Работа с медиаустройствами

Утилитарный хук for работы с аудио/видео устройствами.

**Функции:**

- `getMedia(constraints)` — получение медиапотока с заданными ограничениями
- `getDevices()` — получение списка доступных устройств
- `getDisplayMedia(constraints)` — захват экрана
- `muteAudio(stream)` — включение/выключение аудио в потоке
- `stopStream(stream)` — остановка всех треков в потоке

**Константы:**

- `DEFAULT_AUDIO_CONSTRAINTS` — эхоподавление, автоусиление, шумоподавление
- `DEFAULT_VIDEO_CONSTRAINTS` — 1920x1080, 60fps
- `ADDITIONAL_VIDEO_CONSTRAINTS` — захват окна с курсором

### 4. Компоненты (`components/`)

#### `components/chat/Room.jsx` — Комната голосового чата

Основной компонент для отображения голосовой комнаты.

**Пропсы:**

- `setViewRoom` — функция для закрытия комнаты
- `chatId` — идентификатор чата

**Функционал:**

- Подключается к голосовой комнате через `useVoiceRoom`
- Отображает список участников через `RoomUser`
- Кнопки управления: наушники, микрофон, выход

#### `components/chat/RoomUser.jsx` — Карточка участника

Отображает одного участника голосовой комнаты.

**Пропсы:**

- `participant` — объект участника с полями:
  - `id` — идентификатор
  - `user` — объект пользователя (avatar, username/login)
  - `stream` — MediaStream для воспроизведения аудио
  - `state` — состояние (speaking и др.)

**Функционал:**

- Подключает аудиопоток к `<audio>` элементу
- Автоматическое воспроизведение (autoPlay)
- Отображает аватар и имя пользователя
- Подсвечивает говорящего участника

### 5. Хранилища (`store/`)

#### `store/authStore.jsx` — Аутентификация

Управляет состоянием аутентификации пользователя.

#### `store/chatStore.jsx` — Чаты

Управляет списком чатов и активным чатом.

#### `store/messagesStore.jsx` — Сообщения

Управляет сообщениями в текущем чате.

### 6. Сервисы (`services/`)

Сервисы предоставляют методы для взаимодействия с API:

- `AuthService.jsx` — регистрация, вход, выход
- `ChatService.jsx` — получение списка чатов, создание чатов
- `MessageService.jsx` — отправка, получение, редактирование сообщений
- `UserService.jsx` — получение информации о пользователях

## Архитектура голосовых комнат

### Общая схема работы

```
┌─────────────────┐    WebSocket    ┌─────────────────┐
│   Клиент A      │◄──────────────►│    Сервер       │
│                 │                 │   (Django)      │
│  useVoiceRoom   │                 │                 │
│       +         │                 │  Voice Room     │
│   useWebRTC     │                 │   Manager       │
│                 │                 │                 │
└────────┬────────┘                 └────────┬────────┘
         │                                   │
         │ WebRTC (P2P)                      │
         │                                   │
         ▼                                   ▼
┌─────────────────┐                 ┌─────────────────┐
│   Клиент B      │                 │   Клиент C      │
│                 │                 │                 │
│  useVoiceRoom   │                 │  useVoiceRoom   │
│       +         │                 │       +         │
│   useWebRTC     │                 │   useWebRTC     │
│                 │                 │                 │
└─────────────────┘                 └─────────────────┘
```

### Процесс подключения к комнате

1. **Инициализация**
   - Компонент `Room` монтируется с `chatId`
   - Хук `useVoiceRoom` создает WebSocket соединение
   - Отправляется запрос на присоединение к комнате

2. **Получение состояния комнаты**
   - Сервер отправляет `room_state` с текущими участниками
   - Для каждого участника создается `RTCPeerConnection`
   - Инициализируется локальный аудиопоток

3. **Подключение нового участника**
   - Сервер рассылает `user_joined` всем участникам
   - Существующие участники создают peer connection к новому
   - Отправляют WebRTC `offer` через сервер
   - Новый участник отвечает `answer`
   - Происходит ICE exchange для установления соединения

4. **Обмен медиаданными**
   - После установления P2P соединения потоки передаются напрямую
   - Каждый участник получает аудиопоток от других участников
   - Потоки воспроизводятся через `<audio>` элементы

### Топология сети

**Mesh (полносвязная сеть):**

- Каждый участник соединяется со всеми остальными напрямую
- Для N участников создается N×(N-1)/2 P2P соединений
- Преимущества: низкая задержка, нет единой точки отказа
- Недостатки: нагрузка растет квадратично с количеством участников

**Рекомендуемое ограничение:** до 10-15 участников в комнате

### Обработка ошибок и переподключение

1. **Разрыв WebSocket соединения**
   - Автоматическая попытка переподключения через 3 секунды
   - Сохранение флага `manuallyClosedRef` для различения ручного и аварийного отключения

2. **Heartbeat механизм**
   - Каждые 10 секунд отправляется `ping`
   - Сервер отвечает `pong`
   - Позволяет обнаруживать разрывы соединения

3. **Очистка ресурсов**
   - При отключении закрываются все peer connections
   - Останавливаются все медиапотоки
   - Очищаются таймеры и интервалы

## Безопасность

### Аутентификация

- JWT токены хранятся в localStorage
- Автоматическое обновление токена при истечении
- Refresh token отправляется только на доверенный домен (withCredentials)

### WebRTC

- Используется DTLS-SRTP для шифрования медиапотоков
- STUN сервер для обхода NAT
- В продакшене рекомендуется использовать TURN сервер
